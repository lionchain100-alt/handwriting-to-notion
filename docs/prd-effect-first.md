# 效果优先需求文档

## 1. 文档概述
本文档定义 `handwriting-to-notion` 第二阶段“效果优先”版本的需求。在 MVP 已验证用户需求的前提下，本阶段重点提升识别质量和结构化结果的可用性。

## 2. 背景与目标

### 2.1 背景
MVP 的意义是验证需求，不是做到最强识别。当产品方向被验证后，真正的瓶颈通常会落在 OCR 质量和复杂场景处理能力上。

### 2.2 目标
- 提升复杂手写场景下的识别质量
- 提升结构化结果的可用性和稳定性
- 降低人工修正成本
- 在尽量不改变上层体验的情况下完成底层升级

## 3. 范围

### 3.1 本阶段范围
- 将 OCR 层替换为云 OCR / 文档识别 API
- 保留原有三种输出模式
- 优化图像预处理
- 增强 warning 与低置信度提示
- 支持更复杂的真实输入场景

## 4. 非范围
- 暂不做 Notion API 直连导出
- 暂不做多图合并
- 暂不做强编辑器能力
- 暂不做完整账号系统
- 暂不做大规模历史结果管理能力

## 5. 用户流程
```text
上传图片
→ 图像预处理增强
→ 云 OCR 提取原文
→ LLM 结构化整理
→ 展示结果
→ 用户复制结果
```

## 6. 功能需求

### 6.1 识别质量升级
- 替换 OCR provider 为云 OCR / 文档识别方案
- 兼容复杂手写、混排和手机拍照场景
- 尽可能返回更丰富的中间数据，如置信度、行块信息

### 6.2 输出能力保持一致
- 继续输出 Plain Text、Structured Notes、To-do List
- 保持与 MVP 一致的核心交互方式
- 结果格式继续以 Notion-ready 使用为目标

### 6.3 交互增强
- 增加更明确的低质量提示
- 增加更友好的错误状态
- 可在必要时提示低置信度内容
- 优化复制体验与阅读体验

## 7. 技术方案
- 图像预处理：继续以 `sharp` 为基础
- OCR provider：升级为 Cloud OCR / Document AI
- 结构化输出：继续由 `LLM` 负责
- API / UI 尽量保持不变，仅替换底层 provider
- 工程实现继续遵循 `CLAW.md` 约束

## 8. 验收标准
- OCR 质量明显优于 MVP baseline
- 复杂手写输入场景下的稳定性提升
- 结构化结果满意度提升
- warning 更准确
- 用户人工修正成本下降
- 升级过程中未破坏原有核心流程

## 9. 风险与依赖

### 9.1 风险
- 云 OCR 带来外部依赖和持续调用成本
- 不同 provider 的结果格式差异可能增加兼容复杂度
- 识别层提升后，结构化层可能成为新的瓶颈

### 9.2 应对策略
- 通过 provider 抽象隔离不同 OCR 方案
- 保持输出层和 UI 层稳定
- 继续保留 Plain Text 作为基准输出

### 9.3 依赖
- 云 OCR / 文档识别服务质量
- 成本与调用稳定性
- LLM 重组能力的持续可用性
