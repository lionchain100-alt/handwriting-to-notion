# MVP 收口任务清单

## 1. 文档概述
本文档用于承接 `backlog-mvp.md` 在第一轮 MVP 开发完成之后的收口阶段工作。  
它不再解决“如何把主链路做出来”，而是解决“如何把已经完成的 MVP 主链路收紧为一个可测试、可验证的版本”。

本文档默认基于当前项目状态文件 `PROJECT_STATUS.md`，并遵循 `CLAW.md` 约束执行。

## 2. 当前阶段定义
当前项目已进入：

**MVP 第一轮开发完成，主链路闭环后的收口优化与实测准备阶段。**

当前阶段不再继续扩展第二阶段能力，而是优先完成：
- 残留清理
- 页面与文案收口
- 输出结果可用性收口
- 状态边界收口
- 真实样本测试准备

## 3. 与 `backlog-mvp.md` 的关系
- `backlog-mvp.md`：用于第一轮 MVP 开发任务推进，目标是完成主链路建设
- 本文档：用于第一轮 MVP 开发完成后的收口任务推进，目标是完成质量收口与测试准备

可理解为：
- `backlog-mvp.md` 解决“把 MVP 做出来”
- 本文档解决“把做出来的 MVP 收成可验证版本”

## 4. 收口目标
本轮收口的核心目标如下：

1. 提升页面价值表达清晰度
2. 提升错误提示和 warning 的可理解性
3. 提升三类输出结果的区分度与可用性
4. 收敛状态边界与轻量残留
5. 为真实手写样本测试做好准备

## 5. 本轮非目标
本轮默认不做以下事项：
- 账号系统
- Notion API 直连
- 多图上传 / 多图合并
- 历史记录
- 项目管理能力
- 重型编辑器
- 阶段 2 / 阶段 3 能力提前开发
- 大规模架构重构

## 6. 收口任务清单

### Task C0-01
- **模块**：Content / Entry
- **任务名**：页面文案收口
- **目标**：让首页和工具页从“demo 感”过渡到“可测试工具页”
- **输入**：当前首页与工具页文案
- **输出**：更明确的产品表达文案
- **依赖**：无
- **涉及文件**：
  - `app/page.tsx`
  - `features/output/components/mvp-shell.tsx`
- **验收标准**：
  - 首页和工具页都能清楚表达产品价值
  - 不再出现明显 scaffold / 占位式表达
  - 用户初次进入时能理解产品用途
- **优先级**：P0

### Task C0-02
- **模块**：Empty State / Guidance
- **任务名**：空状态与引导文案收口
- **目标**：让未上传时页面状态完整、清晰、可理解
- **输入**：当前未上传状态页面
- **输出**：完整的空状态与输入引导表达
- **依赖**：C0-01
- **涉及文件**：
  - `features/output/components/mvp-shell.tsx`
  - 如需新增轻量组件，仍应复用现有结构
- **验收标准**：
  - 未上传状态下页面不显空
  - 用户知道支持上传什么内容
  - 用户知道会得到哪些输出结果
- **优先级**：P0

### Task C0-03
- **模块**：Error / Warning
- **任务名**：错误提示与 warning 文案收口
- **目标**：把当前错误信息从工程可读提升为用户可理解
- **输入**：当前错误映射与 warning 展示
- **输出**：更自然、更可行动的错误和 warning 文案
- **依赖**：无
- **涉及文件**：
  - `features/handwriting/services/error-mapping.ts`
  - `features/output/components/output-panel.tsx`
- **验收标准**：
  - OCR / format 失败提示更自然
  - warning 能指导用户优先参考何种输出
  - 错误信息降低理解成本
- **优先级**：P0

### Task C0-04
- **模块**：Restructure Output
- **任务名**：输出结果可用性收口
- **目标**：让 Structured Notes 和 To-do List 与 Plain Text 形成清晰区分
- **输入**：当前 `BasicRestructureProvider` 输出逻辑
- **输出**：更接近 Notion-ready 使用场景的三类结果
- **依赖**：无
- **涉及文件**：
  - `features/handwriting/providers/restructure/basic-restructure-provider.ts`
  - 如需补辅助函数，放入 `features/handwriting/services/`
- **验收标准**：
  - Plain Text 保持原文兜底角色
  - Structured Notes 不只是简单改标题
  - To-do List 不只是机械包一层 checklist
  - 三类结果具有明显用途差异
- **优先级**：P0

### Task C1-01
- **模块**：State Boundary
- **任务名**：状态管理边界收口
- **目标**：梳理上传状态与结果状态的边界，降低状态表达混乱
- **输入**：`UploadPanel` 与 `MvpShell` 当前状态分布
- **输出**：更清晰的状态归属与表达
- **依赖**：C0-01、C0-02
- **涉及文件**：
  - `features/upload/components/upload-panel.tsx`
  - `features/output/components/mvp-shell.tsx`
  - `features/output/components/processing-status.tsx`
  - `features/output/components/output-state.tsx`
- **验收标准**：
  - 上传状态与结果状态边界更清晰
  - 页面不出现重复或冲突状态提示
  - 不引入新的复杂状态系统
- **优先级**：P1

### Task C1-02
- **模块**：Page Entry
- **任务名**：首页与工具页入口关系收口
- **目标**：统一首页和工具页的关系，减少项目脚手架感
- **输入**：当前首页入口形式
- **输出**：更清晰的首页定位
- **依赖**：C0-01
- **涉及文件**：
  - `app/page.tsx`
- **验收标准**：
  - 首页定位清楚
  - 与 `/handwriting-to-notion` 的关系清楚
  - 不再像初始化模板页
- **优先级**：P1

### Task C1-03
- **模块**：Lightweight Cleanup
- **任务名**：轻量残留继续清理
- **目标**：继续去除不参与当前 MVP 收口的目录噪音
- **输入**：当前项目目录结构
- **输出**：更收敛的模块目录结构
- **依赖**：无
- **涉及范围**：
  - 空目录或无实际实现内容的占位模块
- **验收标准**：
  - 不再保留明显“看起来像功能模块、实际上为空”的目录
  - 不误删当前或近期会使用的实现
- **优先级**：P1

### Task C1-04
- **模块**：Test Readiness
- **任务名**：为真实样本测试做准备
- **目标**：让当前 MVP 页面进入适合执行样本回归的状态
- **输入**：收口后的页面、输出与状态表现
- **输出**：可用于样本测试的 MVP 版本
- **依赖**：C0-01、C0-02、C0-03、C0-04
- **涉及范围**：
  - 页面文案
  - warning 提示
  - 输出结果
  - 状态表现
- **验收标准**：
  - 页面已适合进入真实样本测试
  - 没有明显影响测试判断的展示噪音
- **优先级**：P1

## 7. 建议执行顺序
建议按以下顺序执行：

1. `C0-01` 页面文案收口
2. `C0-02` 空状态与引导文案收口
3. `C0-03` 错误提示与 warning 文案收口
4. `C0-04` 输出结果可用性收口
5. `C1-01` 状态管理边界收口
6. `C1-02` 首页与工具页入口关系收口
7. `C1-03` 轻量残留继续清理
8. `C1-04` 为真实样本测试做准备

## 8. 收口完成后的预期状态
本轮收口完成后，项目应进入以下状态：

**MVP 已收口，可正式进入真实手写样本测试阶段。**

届时下一步应切换到：
- 样本测试方案
- 样本回归执行
- 基于测试结果决定第二轮优化方向
