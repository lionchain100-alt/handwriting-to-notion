# MVP 开发任务单 / Sprint Backlog

## 1. 文档概述
本文档基于 `wbs-mvp.md` 继续细化，生成可直接执行的 MVP 开发任务单。任务单用于实际开发推进、优先级排序、依赖识别和验收管理。

## 2. 背景与目标

### 2.1 背景
WBS 解决“要做什么”，Backlog 解决“具体怎么执行”。本文件将 MVP 范围进一步拆成可直接排期和落地的任务项。

### 2.2 目标
- 提供可执行的开发任务单
- 明确每个任务的输入、输出、依赖和验收标准
- 为分批开发和 Sprint 推进提供依据

## 3. 范围
- MVP 阶段的开发任务拆分
- 从项目准备到联调验收的全部主流程任务

## 4. 非范围
- 不覆盖阶段 2 / 阶段 3 的任务
- 不覆盖人员分配与工时估算
- 不替代后续测试报告和发布清单

## 5. 用户流程
```text
上传手写图片
→ OCR 提取原文
→ 结构化生成结果
→ 展示三类输出
→ 用户复制到 Notion
```

## 6. 功能需求
本 Backlog 围绕 MVP 的四层结构展开：
- types
- services
- API
- UI
并补充 integration 与 QA 任务。

## 7. 技术方案
- 图像预处理：`sharp`
- OCR baseline：`Tesseract.js`
- 结构化输出：`LLM`（先允许 mock provider）
- API：Next.js Route Handlers
- 工程约束遵循 `CLAW.md`

## 8. 验收标准
- 每个任务都有明确输入输出与依赖
- 主路径任务能按优先级排序执行
- 任务单可直接用于开发和验收

## 9. 风险与依赖

### 9.1 风险
- 任务粒度不合理导致执行效率低
- 提前进入 UI 开发导致返工
- mock 与真实服务切换成本高

### 9.2 应对策略
- 严格按照优先级执行
- 保持接口抽象稳定
- 主路径先走 mock，再接真实能力

### 9.3 依赖
- `prd-mvp.md`
- `wbs-mvp.md`
- `CLAW.md`
