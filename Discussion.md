# Discussion

## 1. 文档概述
本文档用于沉淀 `handwriting-to-notion` 项目的讨论结论，统一产品定位、技术判断、阶段路线和关键约束。它不是详细 PRD，而是整个项目的共识总纲，用于指导后续方案设计、需求拆解和开发实施。

## 2. 背景与目标

### 2.1 项目背景
`handwriting-to-notion` 面向的是“手写内容整理进 Notion”这一垂直场景。它不应该被做成一个泛 OCR 工具，而应该围绕“手写笔记 → Notion-ready 输出”建立价值。

### 2.2 核心目标
- 把手写笔记图片转换成适合直接进入 Notion 的结构化内容
- 节省用户手动录入和二次整理的时间
- 优先验证用户是否真正需要这一能力
- 在验证需求后逐步升级识别和结构化能力

### 2.3 核心判断
- 值得做
- 应定位成垂直工具站，而不是通用 OCR 工具
- 真正价值在结构化和 Notion-ready 输出，不在原始字符识别本身
- 成功标准应以“用户是否愿意复制或导出到 Notion”为准，而不是单纯 OCR 准确率

## 3. 范围

### 3.1 当前讨论范围
- 产品定位
- 用户价值
- MVP 方向
- 三阶段技术路线
- OCR、结构化、Notion 输出之间的关系
- 风险与升级策略

### 3.2 产品范围
- 识别手写笔记图片
- 输出纯文本、结构化笔记、待办清单
- 支持复制为适合 Notion 的格式
- 后续支持多图合并、场景模式、直接导出到 Notion

## 4. 非范围
- 不做通用 OCR 站
- 不做简单的“上传图片 → 返回原始文本”工具
- 不做 `image to text` 的克隆版
- 不在第一阶段做复杂账号系统、团队协作和重型编辑器
- 不在第一阶段过度追求字符级 OCR 极限准确率

## 5. 用户流程
```text
上传手写图片
→ 图像预处理
→ OCR 提取原始文本
→ LLM / 规则整理结构
→ 输出纯文本 / 结构化笔记 / 待办清单
→ 用户复制到 Notion
→ 后续支持直接导出到 Notion
```

## 6. 功能与产品结论

### 6.1 产品定位
- Handwriting to Notion
- 把手写笔记转换成干净、结构化的 Notion 页面内容
- 输入包括：纸质笔记、课堂笔记、会议记录、白板、便利贴、草稿纸、笔记本页面

### 6.2 目标用户
- 学生
- 知识工作者
- 创作者 / 研究者
- Notion 重度用户

### 6.3 差异化方向
- Notion-ready 格式化输出
- 结构化结果而不是单纯 OCR 原文
- 面向笔记场景的内容清洗和整理
- 从复制到 Notion，逐步升级为直接导出到 Notion

### 6.4 MVP 结论
MVP 的核心流程应包括：
- 上传图片
- 手写识别
- 输出三种结果：
  - 纯文本
  - 结构化笔记
  - 待办清单
- 支持复制为 Markdown / bullet list / checklist

### 6.5 后续阶段方向
- 直接导出到 Notion API
- AI 清洗 / 重组能力增强
- 多图合并成一页
- 场景化模式：课堂笔记、会议笔记、Brainstorm、便利贴
- 人工修正 / 编辑流程
- 更强的版面理解能力

## 7. 技术方案结论

### 7.1 技术原则
这不是纯 OCR 项目，而是：
- 图像预处理
- OCR 原文提取
- 结构理解 / 内容清洗
- Notion-ready 输出

### 7.2 输入特点结论
常见真实输入存在：
- 倾斜 / 歪斜
- 阴影和光照不均
- 模糊
- 透视变形
- 字迹潦草
- 中英混写
- 弱结构内容，如箭头、圈画、缩进、项目符号、checkbox 标记

因此，图像预处理是必要环节，不能省略。

### 7.3 图像预处理结论
推荐预处理步骤：
- Resize
- Grayscale
- Contrast enhancement
- Denoise
- Binarization / adaptive threshold
- Deskew
- Crop
- 可选透视矫正

建议：
- 前端负责预览、旋转、基础裁剪、压缩
- 处理层负责灰度化、阈值化、去噪、纠偏
- MVP 阶段先用 `sharp`

### 7.4 OCR 与结构化结论
MVP 阶段不建议自训练模型。
推荐采用分层设计：
- OCR 负责尽量忠实提取原文
- LLM 负责清洗、重组和输出 Notion-ready 结果

长期原则：
- 识别层和格式化层解耦
- OCR provider 可替换
- 结构化输出层尽量保持稳定

### 7.5 三类方案结论
#### 方案 A：Tesseract.js
适合：
- 低成本 baseline
- MVP 原型验证

不足：
- 对复杂手写、中文手写、混乱排版较弱

#### 方案 B：云 OCR / 文档识别 API
适合：
- 提升识别质量
- 更接近产品可用效果

不足：
- 有成本
- 有第三方依赖
- 仍然需要结构化层

#### 方案 C：多模态模型直接输出
适合：
- 增强结构理解
- 更强的笔记整理和 checklist 提取

不足：
- 有幻觉风险
- 不适合单独承担原文真相来源

### 7.6 最终技术路线结论
采用三阶段递进方案，并在验证用户需求后逐步升级：

#### 阶段 1：低成本验证 MVP
```text
sharp 预处理
→ Tesseract.js OCR
→ LLM 结构化整理
```

#### 阶段 2：效果优先升级
```text
sharp 预处理
→ 云 OCR / 文档识别 API
→ LLM 结构化整理
```

#### 阶段 3：完整版体验增强
```text
image
→ 多模态模型生成结构化结果
→ OCR Plain Text 通道保底
```

### 7.7 架构原则结论
代码设计必须围绕可替换 provider：
- `preprocess provider`
- `ocr provider`
- `restructure provider`
- `notion formatter`

这样可以保证：
- 阶段 1 → 阶段 2 只替换 OCR 层
- 阶段 2 → 阶段 3 增强多模态能力而不破坏整体产品流程

## 8. 验收标准
- 产品定位清晰：始终围绕 handwriting + Notion 场景
- 三阶段路线清晰且升级条件明确
- 技术原则明确：OCR 提取、LLM 重组、输出 Notion-ready
- 核心价值明确：用户愿意复制或导出结果到 Notion
- 文档可作为后续方案、PRD 和开发实施的统一总纲

## 9. 风险与依赖

### 9.1 主要风险
- 项目做成泛 OCR 工具，丧失差异化
- 过度追求 OCR 准确率，忽视最终结构化可用性
- LLM 出现幻觉，导致结果不可信
- 输入图片质量差，导致识别效果波动大
- 没有提前做 provider 抽象，导致后续升级成本高

### 9.2 应对策略
- 全部围绕 handwriting + notes + Notion-ready 输出展开
- 保留 Plain Text 作为原文兜底
- 对结构化层加强 prompt 约束
- 增加 warning / 低置信度提示
- 从第一版开始做 provider 分层设计

### 9.3 依赖
- OCR 引擎能力（Tesseract.js / Cloud OCR）
- LLM 结构化能力
- 后续 Notion API 导出能力
