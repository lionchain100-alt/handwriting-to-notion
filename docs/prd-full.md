# 完整版需求文档

## 1. 文档概述
本文档定义 `handwriting-to-notion` 完整版产品需求。在产品方向已验证的前提下，本阶段目标是打造从上传、识别、整理到导出 Notion 的完整闭环。

## 2. 背景与目标

### 2.1 背景
在需求验证和识别质量提升之后，产品的下一步不是继续堆 OCR，而是补齐完整体验：更强的结构化结果、更好的场景支持、更完整的导出能力，以及更可控的结果校正机制。

### 2.2 目标
- 提供更强的结构化结果质量
- 提供更丰富的场景模式
- 支持直接导出到 Notion
- 支持更复杂、更真实的用户工作流
- 形成从上传到导出的完整产品闭环

## 3. 范围

### 3.1 本阶段范围
- 多模态模型直接参与结构化输出
- OCR Plain Text 通道继续作为保底
- 支持 Notion API 导出
- 支持多图合并
- 支持场景模板
- 支持结果编辑 / 校正
- 增强 warning / confidence 机制

## 4. 非范围
- 不把产品扩展成泛文档平台
- 不做与核心场景无关的大而全协作系统
- 不在本阶段扩展到无关内容生产场景

## 5. 用户流程
```text
上传一张或多张手写图片
→ 系统预处理与识别
→ 多模态模型生成结构化结果
→ OCR 原文作为保底对照
→ 用户查看 / 编辑 / 校正
→ 一键复制或直接导出到 Notion
```

## 6. 功能需求

### 6.1 结构化增强
- 多模态模型参与生成结构化笔记和待办结果
- 输出更适合真实 Notion 使用场景
- 对复杂手写内容、弱结构场景有更强适应力

### 6.2 导出能力
- 支持直接导出到 Notion page / database
- 支持将结果映射为 Notion block 结构
- 导出失败时支持错误提示与重试

### 6.3 场景能力
- 支持 class notes、meeting notes、sticky notes、whiteboard 等模式
- 支持不同场景下的结果模板化输出

### 6.4 编辑与校正
- 用户可查看 OCR 原文与结构化结果
- 用户可对最终结果进行编辑和校正
- 系统可对低置信度内容进行重点提示

### 6.5 多图处理
- 支持多图上传
- 支持多图排序与合并
- 支持多图结果汇总成一个统一输出

## 7. 技术方案
- OCR provider 继续作为原文兜底层
- Multimodal model 作为结构理解增强层
- Notion formatter + export service 负责输出映射与导出
- 通过 provider 编排支持不同阶段能力叠加
- 工程实现继续遵循 `CLAW.md` 的结构和顺序要求

## 8. 验收标准
- 输出质量达到“可直接进入 Notion”的产品标准
- 支持真实复杂手写场景
- 从上传到导出形成完整闭环
- 用户对结构化结果和 Notion 导出能力认可度高
- 系统可在保真、可用性和体验之间取得平衡

## 9. 风险与依赖

### 9.1 风险
- 多模态模型存在幻觉风险
- 功能增多后交互复杂度上升
- Notion API 集成带来更多外部约束
- 多图处理和编辑能力会提高实现复杂度

### 9.2 应对策略
- 始终保留 OCR Plain Text 作为原文兜底
- 对结果不确定内容增加 warning / confidence 提示
- 通过场景模式和模块化设计控制产品复杂度
- 逐步引入导出、编辑、多图能力，而不是一次性堆满

### 9.3 依赖
- 多模态模型效果与稳定性
- Notion API 的接入与限制
- 更复杂的前后端交互与状态管理能力
